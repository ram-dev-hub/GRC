## Domain Overview

The GRC Risk Module is a comprehensive risk management system for organizations to:
- Create, track, and analyze risks across departments
- Score risks by impact (1-5) and likelihood (1-5)
- Implement risk treatments (Accept, Mitigate, Avoid, Transfer)
- Manage risk workflows from creation through closure
- Enforce multi-level approvals and governance
- Track remediation tasks and compliance audit trails

## Core Entities & Models

### Risk
The central entity representing a potential threat or vulnerability to the organization.

**Key Properties**:
- `risk_id`: Auto-generated unique identifier (ABC-DEPT##)
- `description`: 1-500 character narrative
- `owner`: Person responsible for managing the risk
- `department`: IT GRC | Finance | Customer Support | Learning & Development
- `priority`: Highest | High | Medium | Low
- `status`: Open | Closed
- `workflow_state`: Open → UnderReview → ReadyForReview → Approved → FinalClosed → Archived

**Related Data**:
- Impact score (1-5): Financial, Operational, Compliance, Reputational, Security
- Likelihood score (1-5): Based on threat landscape, controls, history, attacker capability, vulnerability
- Risk score (calculated): Impact × Likelihood = 1-25
- Treatment strategy: Accept | Mitigate | Avoid | Transfer
- Remediation tasks (for Mitigate strategy)
- Approvals: Multi-level governance

## Acceptance Criteria Standards

All stories must include 10+ Acceptance Criteria (AC), each following pattern:

**AC Format**:
```
AC [STORY].[NUMBER]: [One sentence describing the criterion]
- Detail 1 about requirement
- Detail 2 about validation
- Detail 3 about error handling
```

**Quality Guidelines**:
- Each AC must be testable and verifiable
- Use technical language (HTTP codes, data types, field names)
- Include both happy path and error cases
- Reference database tables, API endpoints where applicable
- Specify permissions/authorization requirements
- Include audit logging requirements if applicable

**Example**:
```
AC 1.1: Risk ID format matches company standard (ABC-DEPT##)
- ABC = Department code (ITG, FIN, CSU, LAD)
- DEPT = Department identifier
- ## = Sequential number (auto-increment)
- Example: ITG-GRC01, FIN-001, CSU-042
```

Each Story should have 4-7 Tasks that break down implementation work.

**Task Format**:

Task [STORY].[NUMBER].[SUBTASK]: [One sentence summary]

## API Endpoint Conventions

**Base URL**: `/api/v1`

**Methods**:
- `POST` - Create resource (HTTP 201)
- `GET` - Retrieve resource (HTTP 200)
- `PATCH` - Partial update (HTTP 200)
- `PUT` - Full replace (HTTP 200)
- `DELETE` - Delete resource (HTTP 204)

**URL Patterns**:
- Create: `POST /api/v1/risks`
- List: `GET /api/v1/risks?limit=20&offset=0&filters...`
- Get: `GET /api/v1/risks/{riskId}`
- Update: `PATCH /api/v1/risks/{riskId}`
- Delete: `DELETE /api/v1/risks/{riskId}`
- Sub-resource: `PATCH /api/v1/risks/{riskId}/impact`

**Request/Response**:
- Format: JSON
- Headers: Content-Type: application/json, Authorization: Bearer token
- Error format: { error: { code, message, details: [{field, message}] } }

## Database Schema Conventions

**Naming**:
- Tables: `plural_lowercase` (e.g., `risks`, `risk_owners`, `audit_logs`)
- Columns: `snake_case` (e.g., `risk_id`, `created_at`, `modified_by`)
- Primary Key: `id SERIAL PRIMARY KEY`
- Foreign Keys: `{table_singular}_id` (e.g., `owner_id`, `risk_id`)

**Common Columns**:
- `created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP`
- `modified_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP`
- `created_by VARCHAR(255)` (email)
- `modified_by VARCHAR(255)` (email)
- `deleted_at TIMESTAMP NULL` (for soft deletes)

**Constraints**:
- `PRIMARY KEY` on id
- `UNIQUE` on external identifiers
- `CHECK` constraints for enums
- `REFERENCES` for foreign keys
- `NOT NULL` for required fields

**Indexes**:
- All foreign keys
- All filter/sort fields
- Frequently used combinations

## Workflow State Machine

**Valid Transitions**:
```
Open
├─→ UnderReview
│   ├─→ ReadyForReview
│   │   ├─→ Approved
│   │   │   └─→ FinalClosed
│   │   │       └─→ Archived
│   │   └─→ UnderReview (rework)
│   └─→ Open (need more data)
├─→ FinalClosed
│   └─→ Archived
└─→ Archived (emergency path)
```

**Permissions**:
- Risk Officer: Most transitions
- Risk Manager: Approve → Approved
- Department Head: Approve → FinalClosed
- Admin: Override any transition

## Treatment Strategy Workflows

**ACCEPT**: Acknowledge and live with the risk
- Requires executive approval
- Sets acceptance deadline
- Risk moves to Approved state

**MITIGATE**: Reduce risk through controls
- Creates remediation tasks
- Tracks progress to residual score target
- Tasks must be completed before closure

**AVOID**: Eliminate activity causing risk
- Requires business unit approval
- Document alternative approach
- Risk can be archived after avoidance

**TRANSFER**: Shift risk to third party
- Track insurance or outsourcing details
- Monitor third-party compliance
- Risk remains under observation

## Scoring Framework

### Impact Score (1-5)
1. **Minimal/Negligible** - <$10K loss, minimal disruption
2. **Minor/Limited** - $10-50K loss, limited impact
3. **Moderate/Significant** - $50-500K loss, significant impact
4. **Major/Critical** - $500K-5M loss, critical impact
5. **Catastrophic/Severe** - >$5M loss, business-threatening

### Likelihood Score (1-5)
1. **Rare** - <1% annual probability
2. **Unlikely** - 1-10% annual probability
3. **Possible** - 10-50% annual probability
4. **Likely** - 50-90% annual probability
5. **Almost Certain** - >90% annual probability

### Risk Score Severity
- **Low Risk** (1-5): Green badge
- **Medium Risk** (6-12): Yellow badge
- **High Risk** (13-18): Orange badge
- **Critical Risk** (19-25): Red badge

---

## Approval Workflow

**By Severity**:
- **Critical (19-25)**: CRO + 2 Risk Council members
- **High (13-18)**: Risk Manager + Department Head
- **Medium (6-12)**: Department Head only
- **Low (1-5)**: Auto-approve by Risk Officer

**Special Cases**:
- Security risks (security_impact ≥ 4): Require Security Lead approval
- Compliance risks (compliance_impact ≥ 4): Require Compliance Officer approval
- Cross-department risks: All affected Department Heads must approve
- Emergency risks: CRO can fast-track (retrospective approval within 5 days)

## Audit Trail Requirements

Every modification must be logged with:
- `action`: What changed (Risk Created, Impact Score Updated, etc.)
- `actor_email`: Who made the change
- `timestamp`: When (UTC)
- `old_values`: Previous values (JSONB)
- `new_values`: New values (JSONB)
- `changed_fields`: Array of field names that changed
- `ip_address`: Source IP
- `user_agent`: Browser info (if applicable)

**Audit Retention**: 7 years minimum

---

## Authorization Model

**Roles**:
- **Risk Officer**: Create, update risks; perform analysis; not approve
- **Risk Manager**: Risk Officer privileges + can approve High risks
- **Department Head**: Can approve Medium risks in own department
- **Admin**: Full access, can override any rule
- **Auditor**: Read-only access to all risks and audit trails
- **CRO**: Chief Risk Officer, approves Critical risks

**Principle**: Users can only view/modify risks in their department or department they have explicit permission for

## Data Validation Rules

**Risk Description**:
- Required, min 1 char, max 500 chars
- Cannot be all numbers/special characters
- Case-sensitive

**Risk Owner**:
- Required field
- Must exist in risk_owners table
- Valid email format

**Priority**:
- Required field
- Enum: Highest | High | Medium | Low (case-sensitive)

**Department**:
- Required field
- Enum: IT GRC | Finance | Customer Support | Learning & Development
- Read-only after creation (cannot change department)

**Scores (Impact/Likelihood)**:
- Numeric, 1-5 range
- Justification required (max 500 chars)
- Cannot be calculated automatically (must be manually entered)

**Dates**:
- ISO 8601 format (YYYY-MM-DDTHH:mm:ssZ)
- UTC timezone
- created_at: Set automatically, cannot be updated
- due_date: Must be future date (for tasks)

## Error Handling

**HTTP Status Codes**:
- `200 OK` - Successful GET, PATCH, PUT
- `201 Created` - Successful POST
- `204 No Content` - Successful DELETE
- `400 Bad Request` - Validation error (invalid format, missing required field)
- `403 Forbidden` - Insufficient permissions
- `404 Not Found` - Resource not found
- `409 Conflict` - Concurrent modification (ETag mismatch)
- `422 Unprocessable Entity` - Validation error (invalid value for field, invalid state transition)

**Error Response Format**:
```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Validation failed",
    "details": [
      { "field": "impact_score", "message": "must be between 1 and 5" }
    ]
  }
}
```

2. Specify permissions for each transition
3. List business rules that must be checked
4. Define notifications sent
