## Key Concepts to Remember

### Risk Scoring
- Impact Score (1-5): Based on financial, operational, compliance, reputational, security impact
- Likelihood Score (1-5): Based on threat landscape, control effectiveness, incident history, attacker capability, vulnerability status
- Risk Score (calculated): Impact × Likelihood = 1-25
- Severity Level (derived): Low (1-5), Medium (6-12), High (13-18), Critical (19-25)

### Workflow States
- **Open**: New risk, awaiting analysis
- **UnderReview**: Analysis in progress
- **ReadyForReview**: Ready for stakeholder review
- **Approved**: Approved for treatment
- **FinalClosed**: Treatment complete
- **Archived**: Historical record

### Approval Levels
- **Critical (19-25)**: CRO + 2 Risk Council members
- **High (13-18)**: Risk Manager + Department Head
- **Medium (6-12)**: Department Head
- **Low (1-5)**: Risk Officer (auto-approve)

### Treatment Strategies
- **Accept**: Acknowledge and live with the risk
- **Mitigate**: Reduce risk through controls/remediation tasks
- **Avoid**: Eliminate activity causing the risk
- **Transfer**: Shift risk to insurance or outsourcing

### Audit Trail
Every change must log: action, actor_email, old_values, new_values, changed_fields, timestamp, ip_address

### Database Naming
- Tables: `plural_lowercase` (risks, risk_owners, audit_logs)
- Columns: `snake_case` (risk_id, impact_score, created_at)
- Primary Key: `id SERIAL PRIMARY KEY`
- Foreign Keys: `{table_singular}_id` (owner_id, risk_id)

### API Patterns
- Create: POST /api/v1/{resource}
- List: GET /api/v1/{resource}?filters
- Get: GET /api/v1/{resource}/{id}
- Update: PATCH /api/v1/{resource}/{id}
- Delete: DELETE /api/v1/{resource}/{id}
- Sub-resource: PATCH /api/v1/{resource}/{id}/{action}

## Questions to Ask Yourself

When reviewing specifications:
- Are all ACs testable by a QA engineer?
- Can a backend engineer implement each AC independently?
- Can a frontend engineer build the UI from the specification?
- Did I document what happens if a request fails (400, 403, 404, 422)?
- Did I include audit logging requirements?
- Did I specify authorization/permissions?
- Did I reference database schema and field names?
- Are the error messages user-friendly and helpful?
- Did I avoid ambiguity (e.g., "sometimes" vs. "when X condition")?
- Did I consider edge cases (concurrent requests, invalid data, missing fields)?
